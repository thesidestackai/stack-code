## 9. Command Deviation Note — apply-bundle generator vs plan apply executor

```text
The S2E apply-gate prompt(s) named `claw plan apply-bundle <preview-generator-result.json> <approval-result.json>`
as "the apply command" and expected it to write the target (before→after).

Runtime evidence clarified that `claw plan apply-bundle` is the apply-bundle GENERATOR:
  - it validates the approval-result and ASSEMBLES apply-bundle.json
  - schema a2-l2b-apply-bundle-generator-result.v1; markers a2-l2b-approval-result-validated / a2-l2b-apply-bundle-created
  - it does NOT write the target (target stayed at before_sha after it ran)

The actual target-writing EXECUTOR is:
  claw plan apply <apply-bundle.json>
  - schema a2-l2b-apply-result.v1; outcome "applied"; marker a2-l2b-write-applied

Resolution: the generator was run first (prior lane) to produce apply-bundle.json; then, in the S2E
execution lane, the apply-bundle and every hash/approval binding were REVALIDATED, and exactly one
`claw plan apply <apply-bundle.json>` was run to write the target before→after. The executor was the
correct command to fulfill the gate's stated Objective and before→after transition.

Follow-up doc/test correction (optional, §12): make the S2E prompt/docs name `plan apply <apply-bundle.json>`
as the target-writing step, with `apply-bundle` clearly labeled the generator.
```
